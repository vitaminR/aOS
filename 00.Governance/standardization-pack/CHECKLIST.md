# Standards Submission Checklist

Use this as the end-to-end tracker for getting the `{a}OS` model ready for standards engagement.

## Phase 0 — Core decisions

- [x] Decide whether `{a}OS` stays **brand-only** while the standards title stays neutral
- [x] Pick the working neutral title → **Agentic Systems — Reference Model and Vocabulary**
- [x] Decide what is actually being proposed:
  - [x] reference model + terminology + classification schema
- [x] Decide primary first route:
  - [x] IEEE SA (with parallel NIST alignment)
- [x] Reference model written (v1.0)
- [x] Vocabulary locked (Stratum, Axis, Substrate, Construct, Primitive)
- [x] Interactive Explorer built with 40+ products classified
- [x] NIST AI RMF crosswalk started (draft)
- [x] Brand vs. neutral naming decision made
- [x] Dual-license structure established (CC BY 4.0 + Apache 2.0)
- [x] Atlas license addendum documented

## Phase 1 — Create the standards-ready package

- [ ] Write a 1-page problem statement
- [ ] Write scope and non-scope
- [ ] Define each official term once, cleanly
- [ ] Add plain-language aliases where helpful
- [ ] Create one canonical hierarchy diagram
- [ ] Write 2–3 worked use cases
- [ ] Document why existing frameworks do not already solve the exact problem

## Phase 2 — Clean neutral spec

- [ ] Remove product-marketing language from standards draft
- [ ] Remove implementation-specific UI language from standards draft
- [ ] Separate:
  - [ ] brand terms
  - [ ] formal terms
  - [ ] plain-language teaching aliases
- [ ] State whether the model is descriptive, prescriptive, or both
- [ ] State whether the model is intended for:
  - [ ] classification
  - [ ] interoperability
  - [ ] governance mapping
  - [ ] architecture communication

## Phase 3 — Evidence and reviewer support

- [ ] Recruit 5–10 outside reviewers
- [ ] Capture objections and terminology pain points
- [ ] Record where reviewers agree the model adds value
- [ ] Build a short evidence memo showing real demand / use cases

## Phase 4 — AI RMF crosswalk

- [ ] Map `{a}OS` strata and axes to NIST AI RMF concepts
- [ ] Identify where `{a}OS` aligns strongly
- [ ] Identify where `{a}OS` extends beyond AI RMF
- [ ] Identify where there is no direct mapping
- [ ] Add notes and rationale for every mapping claim
- [ ] Keep the crosswalk conservative — no fake alignment

## Phase 5 — IEEE path

- [ ] Review IEEE standards development overview
- [ ] Review NesCom PAR submitter guidance
- [ ] Decide whether to start with community/pre-standardization outreach first
- [ ] Create IEEE account if needed
- [ ] Identify sponsoring standards committee / organizational home
- [ ] Draft PAR inputs:
  - [ ] scope
  - [ ] purpose
  - [ ] need
  - [ ] stakeholders
- [ ] Confirm who will be the submitter / project lead
- [ ] Submit only after neutral scope is stable

## Phase 6 — ISO / ANSI path

- [ ] Review ISO “Get involved” route
- [ ] Confirm national standards body path
- [ ] If U.S.-based, contact ANSI / relevant mirror committee route
- [ ] Identify best home:
  - [ ] ISO/IEC JTC 1/SC 42 WG 1 (Foundational standards)
  - [ ] other liaison / vocabulary route if advised
- [ ] Prepare a new-work / concept package suitable for committee discussion
- [ ] Confirm whether the proposal should be:
  - [ ] new work item
  - [ ] workshop agreement style precursor
  - [ ] liaison / discussion paper first

## Phase 7 — Submission-day readiness

- [ ] Final neutral title selected
- [ ] Scope is one page or less and unambiguous
- [ ] Need statement is concrete and market-facing
- [ ] Terminology sheet is internally consistent
- [ ] Diagram is committee-friendly
- [ ] Crosswalk is complete enough to defend value
- [ ] Outreach/contact draft is ready
- [ ] Submission links/accounts are confirmed

## Phase 1 — Atlas Adoption

- [ ] Perpetual license agreement signed with Atlas
- [ ] Pilot deployment on one Atlas workflow
- [ ] Internal validation feedback collected
- [ ] Model applied to real Atlas agentic architecture decisions
- [ ] At least one architecture decision explicitly uses the {a}OS model
- [ ] Pilot metrics captured (cycle time, hours saved, cost per task)

---

## Definition of done for first external outreach

The package is ready for serious outreach when it includes:

- a neutral title
- a concise scope
- a clear need statement
- a terminology sheet
- one architecture/reference-model diagram
- one AI RMF crosswalk draft
- one outreach email / concept note
